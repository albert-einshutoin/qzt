# QZT v0.1 Benchmark Report (2026-07)

Measured on commit `75aa77bb3f2ed6afe8b7d524258e303b7a059930` on
2026-07-19 JST. These results are reproducible local evidence, not an SLA.

## TL;DR

- Hot in-process rare-token QZI lookup was 9–11 µs p50; missing was 1 µs.
- A capped common n-gram query reached 14.6–16.5 ms p50 on the 100 MB matrix.
- QZI token sidecars were 0.455–0.568x source; n-gram sidecars were 0.860–1.161x.
- QZT, reference bytes, ripgrep, and SQLite FTS5 reported count 504 in one smoke.
- Cross-tool search latency, Tantivy, Lucene, and seekable-zstd remain unmeasured.

## Environment

- Mac mini (Mac16,10), Apple M4, 10 cores, 32 GB RAM
- Internal Apple Fabric SSD, APFS; AC power
- macOS 26.5 (25F71), arm64
- `rustc 1.96.0`, `cargo 1.96.0`, ripgrep 15.1.0, SQLite 3.51.0,
  zstd CLI 1.5.7
- Full snapshot: [raw/2026-07-v0.1/environment.txt](raw/2026-07-v0.1/environment.txt)

The commands ran serially on an interactive developer machine. Background OS
services were not disabled. This report therefore publishes ranges and raw
runs instead of treating one local timing as a production guarantee.

## Methodology

The competitive smoke used the deterministic Phase23 C2 generator. The
release matrix used `tests/release_hardening.rs::build_profile_corpus`, while
`make bench-release` used `src/benchmark.rs::release_corpus`; both are separate
deterministic synthetic generators. The matrix covered 1, 10, and 100 MB for
ASCII, UTF-8-mixed, and Japanese text. Each query case used 20 warmups followed
by 500 measured repetitions; the harness sorted those samples and reported
p50/p95/p99.

Three complete runs were retained. A third initial matrix run exceeded 20%
tail variance for the 100 MB UTF-8-mixed common n-gram p99, so the complete
[triggering run](raw/2026-07-v0.1/profile-matrix-retry-trigger-run-3.log) was
retained and the matrix was measured again. The replacement remains run 3;
the table still exposes the first run's 24.438 ms tail rather than hiding it.

The competitive smoke used 65,536 C2 source bytes, 4 KiB QZT chunks, and a
1,024-byte requested range. Each separate process timed each path once over
in-memory buffers; cache state was not controlled. QZT reader open/parsing
happened before its timer, while raw zstd's `decode_all` timer included decoder
construction. The harness records QZT's returned range length, not its physical
decoded bytes. It checks only reported-count correctness for QZT, a reference
byte scan, ripgrep, and SQLite FTS5; it does **not** time the external tools.
QZI is a pre-built index while ripgrep scans the source, so claiming a latency
winner without build and storage cost would be misleading.

## Results

### 1. Range restore: QZT vs whole-file zstd

| Run | QZT 1 KiB range | whole-file zstd | QZT returned / zstd decoded |
| --- | ---: | ---: | ---: |
| 1 | 123 µs | 105 µs | 1,024 / 65,536 |
| 2 | 52 µs | 26 µs | 1,024 / 65,536 |
| 3 | 16 µs | 22 µs | 1,024 / 65,536 |

Only run 3 favored QZT; runs 1 and 2 favored whole-file zstd. The corpus is
only 64 KiB, cache state is uncontrolled, and initialization boundaries are
asymmetric as described above, so this smoke does not establish a general
speedup. The byte
column deliberately does not compare physical decode work: QZT returns 1 KiB
after decoding each overlapping 4 KiB chunk, but the harness does not measure
that decoded amount; raw zstd materializes all 64 KiB. Raw logs:
[run 1](raw/2026-07-v0.1/bench-compete-run-1.log),
[run 2](raw/2026-07-v0.1/bench-compete-run-2.log), and
[run 3](raw/2026-07-v0.1/bench-compete-run-3.log).

### 2. Query latency

All values below are the range across the three retained runs, in
microseconds. Rare-token p50 was 9–11 µs across all 27 cases; missing-token
p50 was 1 µs. These are hot in-process calls after 20 warmups over already-open
container and QZI objects; CLI startup, container/sidecar open, and index build
are excluded. The common n-gram is intentionally high-frequency and capped
before candidate decoding.

| Corpus | Kind | common p50 | common p95 | common p99 |
| --- | --- | ---: | ---: | ---: |
| 1 MB | ASCII | 149–160 | 172–191 | 186–202 |
| 1 MB | UTF-8 mixed | 145–155 | 158–180 | 162–188 |
| 1 MB | Japanese | 148–154 | 175–191 | 185–207 |
| 10 MB | ASCII | 1,457–1,525 | 1,671–1,801 | 1,764–1,875 |
| 10 MB | UTF-8 mixed | 1,342–1,404 | 1,513–1,649 | 1,549–1,815 |
| 10 MB | Japanese | 1,404–1,493 | 1,490–1,768 | 1,564–1,854 |
| 100 MB | ASCII | 16,105–16,503 | 18,668–19,404 | 19,795–21,249 |
| 100 MB | UTF-8 mixed | 14,630–15,393 | 17,261–17,722 | 18,693–24,438 |
| 100 MB | Japanese | 15,484–16,418 | 17,755–18,633 | 19,337–21,723 |

The UTF-8-mixed 100 MB p99 range includes the tail that triggered the retry.
The retry measured 18.737 ms, confirming that p50 was stable while local p99
was sensitive to system noise.

### 3. Search correctness cross-check

| Run | QZT token | Reference bytes | ripgrep | SQLite FTS5 |
| --- | ---: | ---: | ---: | ---: |
| 1 | 504 | 504 | 504 | 504 |
| 2 | 504 | 504 | 504 | 504 |
| 3 | 504 | 504 | 504 | 504 |

This proves reported-count agreement for one fixed-string/token query on the
C2 smoke corpus. QZT, the reference scan, and ripgrep count occurrences;
SQLite counts matching imported rows. C2 has one `qzt` occurrence per matching
row, so all four report 504. This does not prove equivalent query semantics in
general: ripgrep is substring search, SQLite FTS5 tokenizes rows, and QZT v0.1
token search is co-occurrence search with original-byte verification.

The formerly documented `--features bench-compete` command did not enable the
private benchmark modules and failed to compile. Its
[failure log](raw/2026-07-v0.1/bench-compete-failed-documented-command.log) is
retained; the corrected `--all-features` command is used below.

## Honest Reading

QZT's strongest evidence in these runs is candidate selectivity: rare and
missing tokens avoid a raw scan. QZT range restore is designed to decode only
overlapping chunks, but this report's range harness records returned bytes,
not physical decoded bytes, so it does not quantify that advantage. Both
properties matter when an immutable source must remain byte-verifiable and
only a small part is usually disclosed.

The costs deserve equal weight:

1. **Index storage:** the ASCII n-gram sidecar was 1.130–1.161x the source.
   QZT plus QZI can therefore consume more space than the original text.
2. **build-time memory and time:** v0.1 sidecar construction retains the full
   term dictionary and posting map. This report did not measure peak RSS or
   index-build latency, so it cannot claim a total-cost win.
3. **No phrase search:** multi-token search is co-occurrence, not adjacency.
   Normalization, ranking, and mutable updates are also absent.
4. **Asymmetric comparison:** QZI is a pre-built index; ripgrep is an
   unindexed scan. This harness checks cross-tool correctness but does not
   publish cross-tool latency.
5. **Small asymmetric range smoke:** the only direct raw-zstd timing uses 64
   KiB. Results reverse across uncontrolled runs; QZT open is excluded while
   raw decoder construction is included, so there is no universal speed claim.
6. **Coverage gaps:** Tantivy, Lucene, seekable-zstd, real production logs,
   sustained concurrency, and failure recovery remain unmeasured.

### When not to use QZT

Do not use QZT as a full-text database when mutable ranking, language-aware
normalization, phrase queries, frequent updates, or the smallest possible
index is the primary requirement. Do not add QZI when whole-file decompression
is already cheap and selective disclosure is rare. Prefer raw zstd for tiny
files when container verification and partial access do not justify metadata
overhead. This extends the guidance in
[QZT v0.1 Competitive Benchmark Harness](../QZT_v0.1_Competitive_Benchmarks.md).

## Reproduce

Run from repository commit
`75aa77bb3f2ed6afe8b7d524258e303b7a059930` with the versions above:

```sh
make bench-release

QZT_RELEASE_BENCH_QUERY_REPETITIONS=500 \
QZT_RELEASE_BENCH_QUERY_WARMUP_REPETITIONS=20 \
make bench-profile-matrix

cargo test --release --all-features \
  --test phase18_competitive_benchmark -- --nocapture
```

Run each command three times. `--all-features` enables both `bench-compete`
and the internal test surface used by the competitive harness. Compare timing
ranges rather than requiring machine-specific thresholds.
